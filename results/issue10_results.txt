===========================================================================
FINDINGS
===========================================================================

1. PYTHON ROUND-TRIP:
   - Timezone: PRESERVED (does not convert to UTC)
   - Milliseconds (.000): STRIPPED (removed if all zeros)
   - Microseconds (.123456): PRESERVED
   - Z notation: converted to +00:00 (equivalent, just different format)

2. JAVA (from prior observation, not retested):
   - Timezone: CONVERTS to UTC/system timezone
   - This means same moment in time, different representation
   - Cannot re-verify until Issue #1 is resolved

3. BOTH BEHAVIORS ARE SPEC-VALID:
   - xsd:dateTime allows any valid timezone representation
   - Converting to UTC and preserving original are both valid
   - Only matters for exact string comparison (data is equivalent)

4. SEVERITY: LOW
   - No data loss (same moment in time)
   - Only affects string-level comparison of outputs
