===========================================================================
KEY FINDINGS
===========================================================================

1. NO VARIANT WORKS END-TO-END THROUGH JAVA API
   - V2 and V6 pass schema validation but fail at ProvToolbox deserialization
   - The preprocessing (main flow) only adds @id, does NOT copy prefix to document root
   - ProvToolbox needs prefix at document root to resolve namespaces

2. JAN'S TWO TEST FORMATS ARE INCOMPATIBLE WITH EACH OTHER
   - V6 (schema test): prefix inside bundle → passes schema, fails ProvToolbox
   - V7 (deserializer test): prefix at top level → fails schema, passes ProvToolbox
   - These tests pass individually but the formats are mutually exclusive in the API

3. PYTHON ROUND-TRIP BEHAVIOR
   - Python always copies prefix into the bundle on output (even if input had it only at top)
   - Python strips prov/xsd from output (treats as implicit)
   - Python rejects @id inside bundles (V8 fails)

4. THE "BOTH PLACES" FORMAT (V3) would work for Python + ProvToolbox deserialization,
   but fails Java schema validation (schema rejects top-level prefix)

5. Jan's meeting comment "only bundle identifier prefix needs to be inside":
   - V4 tests this exactly (only "ex" inside, prov/xsd at top level)
   - Java schema REJECTS V4 (because prov/xsd at top level are not allowed)
   - So this only works if schema is modified to allow non-bundle prefixes at top level

===========================================================================
WHAT WOULD NEED TO CHANGE FOR EACH APPROACH
===========================================================================

APPROACH A: Accept prefix at top level (match spec/Python)
  → Change Java schema: allow "prefix" at top level
  → No changes needed to preprocessing
  → V1 and V7 format would work
  → Python output would be directly readable

APPROACH B: Keep prefix inside bundle only (current Java schema)
  → Fix preprocessing: copy prefix from inside bundle to document root before ProvToolbox
  → V2 and V6 format would work
  → Python output still needs transformation (Python puts prefix at top level)

APPROACH C: Accept both (most flexible)
  → Change Java schema: allow optional "prefix" at top level
  → Fix preprocessing: if prefix only inside bundle, copy to document root
  → V1, V2, V3, V4, V6, V7 would all work
